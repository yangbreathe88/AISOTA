G4Splat 디버그 분석

야간 RunPod 실험 (4클립 × A100 80GB) — 1클립 성공 / 3클립 실패 원인 분석

실패 발생 지점

G4Splat의 전체 학습은 4 stage로 구성됨. ver_2/3/4가 중단된 위치는 모두 Stage 1(첫 학습)의 초반이며, Stage 2~4는 진입하지 못함.

Stage 0
통과 — MASt3R SfM → align_charts → render_chart_views → plane_excavator → plane_refine_depth
Stage 1
OOM — refine_free_gaussians.py → train_with_refine_depth.py:560 → gaussian_model.py:633 densify_and_prune → densify_and_split에서 메모리 초과 (ver_2/3은 600 iter 시점, ver_4는 0 iter optimizer.step() 직후)
Stage 2
미도달 — see3d_inpaint #1 + 2차 학습 (See3D weights 부재로 어차피 실패할 단계)
Stage 3
미도달 — see3d_inpaint #2 + 3차 학습
Stage 4
미도달 — see3d_inpaint #3 + 4차 학습 + mesh extract

가우시안 점 수 폭증 추적

단계 ver_1 (성공) ver_2 (실패) 분석
SfM points.ply 1.5M pts (38MB) 미상 (다운로드 전 중단) 시작 SfM 점 수가 영상마다 상이
input.ply (학습 시작) 1.5M pts 더 컸을 가능성 높음 SfM 결과를 그대로 학습 input으로 사용
iter 600 (densify 중) 26.5M pts 45.4M pts (1.7×) 동일 시점 ver_2가 1.7배 dense → 폭증
iter 7000 (학습 종료) 2.3M pts (안정) — (중단) ver_1은 prune으로 정리됨

근본 원인 (Root Cause)

  1. SfM 단계 점 수가 영상마다 상이 — ver_2/3/4 영상의 텍스처가 ver_1보다 풍부(또는 노이즈)하여 MASt3R가 더 많은 매칭점을 생성함.
  2. G4Splat의 default densify hyperparam이 자체 데이터에 과도하게 공격적:
    • densify_grad_threshold = 0.0002 — 임계값이 낮아 clone/split이 빈번하게 발생
    • densify_until_iter = 3500 + densification_interval = 100 → 학습당 약 30회 densify 호출
  3. 점 폭증 메커니즘:
    • densify_and_clone: grad가 큰 작은 점 복제 (1→1)
    • densify_and_split: grad가 큰 큰 점 분할 (1→2)
    • opacity_reset으로 정리되나 점 증가 속도가 이를 상회 → A100 80GB도 한계 초과

데모 데이터 vs 자체 데이터

데모 영상은 ScanNet++/Replica 등 이상적 데이터로, 정적 씬·orbit 카메라 동선·균일 조명·수동 노출/초점 고정·반복 텍스처 부재 조건을 갖춤. 자체 영상은 1080p 30fps 핸드폰 영상으로 panning 동선·자동 노출/AF·모션 블러·반복 패턴 가능성을 포함함. G4Splat의 MASt3R SfM이 후자의 데이터 분포에 취약하여 잘못된 매칭 → 점 폭증 → OOM으로 이어짐. 스튜디오 설정 오류가 아니라 가우시안 분야 공통의 데이터-품질 의존성 문제임.

리타게팅 분석 (모션 → MetaHuman)

AI 모션 출력(Kimodo·PromptHMR·GVHMR)의 공통 블로커 — source skeleton → MetaHuman body 적용 단계 분석

문제 정의

source skeleton(SMPL/HumanIK/Kimodo)의 모션을 target skeleton(MetaHuman body)에 적용하는 단계. 본의 의미가 매칭되더라도 다음 차이로 인해 정합이 어긋남: rest pose 차이(회전 delta 어긋남), 본 roll 차이(twist 어긋남), 본 길이 차이(IK 보정 필요), 축 컨벤션 차이(forward 방향 반전 등). 슈트 캡쳐(FBX)만 이 단계를 우회할 수 있음.

Rest pose 갭

자세어깨 → 손 각도사용 도구MH 대비 갭
T-pose0° (수평)SMPL · 일부 mocap—
SMPL 출력약 15° 아래PromptHMR · GVHMR약 30°
A-pose (기준)약 45° 아래MetaHuman0°
I-pose약 75° 아래 (거의 수직)Kimodo약 60°

갭이 클수록 자동 정합이 어려움. SMPL → MH는 약 30°, Kimodo → MH는 약 60°.

시도한 워크플로 (Blender + Rokoko + Python)

  1. Source BVH(SMPL/HumanIK) + Target MH body FBX 임포트
  2. Python 스크립트로 본 페어 22개 자동 매핑 (case-insensitive)
  3. Rokoko rest pose 보정 옵션 활성화 + Copy Rotation(WORLD) constraint + Apply Pose as Rest
  4. Source axis flip(180° yaw) 적용 후 bpy.ops.rsl.retarget_animation() 베이크
  5. 2차 Bake Action(Visual Keying + Clear Constraints)으로 잔여 constraint 제거 → FBX export → UE 임포트

실패 원인

  • 본 roll(twist axis)이 두 스켈레톤 간 상이 → world rotation을 매치해도 local twist가 어긋남
  • 부모 본 회전 적용 시 자식 본 visual 위치가 변동 → 자식 본 정합이 재차 어긋남 (체인 누적 오차)
  • MH의 twist·correction 본을 매핑하지 않아 rest에 남는데, parent rest 변경 시 visual 결과가 왜곡됨
  • 매 alignment 후 다른 부위가 어긋나 수렴하지 않음
결론: 무료 Blender 솔루션으로 SMPL(약 15° 갭)은 부분적으로 동작하나, Kimodo(약 60° 갭)는 수렴 불가. 본 roll·축 컨벤션 차이가 누적되어 무료 자동화 경로로는 한계가 명확함.

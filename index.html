🔬 G4Splat 디버그 보고서

야간 RunPod 실험 (4영상 × A100 80GB) — 1개 성공 / 3개 실패 원인 분석

📍정확히 어디서 죽었나

G4Splat 의 전체 학습은 4 stage로 구성됨. ver_2/3/4 가 죽은 위치는 모두 Stage 1 (첫 학습) 의 초반이고, Stage 2~4 는 시작도 못 함.

Stage 0
✅ MASt3R SfM → align_charts → render_chart_views → plane_excavator → plane_refine_depth — 전부 통과
Stage 1
❌ refine_free_gaussians.py → train_with_refine_depth.py:560 → gaussian_model.py:633 densify_and_prune → densify_and_split 에서 OOM (ver_2/3 의 경우 600 iter 시점, ver_4 는 0 iter 의 optimizer.step() 즉시 죽음)
Stage 2
⏸ see3d_inpaint #1 + 2차 학습 — 도달 못 함 (또한 See3D weights 도 없어서 어차피 실패할 단계)
Stage 3
⏸ see3d_inpaint #2 + 3차 학습
Stage 4
⏸ see3d_inpaint #3 + 4차 학습 + mesh extract

📊점 수 폭증 데이터 (가우시안 개수 추적)

단계 ver_1 (성공) ver_2 (실패) 분석
SfM points.ply 1.5M pts (38MB) 모름 (다운로드 전 죽음) 시작 SfM 점 수가 영상마다 다름
input.ply (학습 시작) 1.5M pts 아마 더 컸을 듯 SfM 결과를 그대로 학습 input 으로 사용
iter 600 (densify 중) 26.5M pts 45.4M pts (1.7×) ver_2 가 동일 시점 1.7배 더 dense → 폭증
iter 7000 (학습 끝) 2.3M pts (안정) — (죽음) ver_1 은 prune 으로 정리됨

🔑근본 원인 (Root Cause)

  1. SfM 단계 점 수가 영상마다 천차만별 — ver_2/3/4 영상의 텍스처가 ver_1 보다 풍부(또는 노이즈)해서 더 많은 매칭점 생성
  2. G4Splat default densify hyperparam 이 우리 데이터에 너무 공격적:
    • densify_grad_threshold = 0.0002 ← 너무 낮음 (clone/split 자주 발생)
    • densify_until_iter = 3500 + densification_interval = 100 → 매 학습마다 30번 densify 호출
  3. 점 폭증 메커니즘:
    • densify_and_clone: grad 큰 작은 점 복제 (1→1)
    • densify_and_split: grad 큰 큰 점 분할 (1→2)
    • opacity_reset 으로 정리하지만, 그보다 빠르게 점 증가 → A100 80GB 도 못 견딤

🧬왜 데모는 잘 되는데 우리는 안 되나

데모 영상은 ScanNet++/Replica 같은 이상적 데이터: 정적 씬, orbit 카메라 동선, 균일 조명, 수동 노출/초점 고정, 반복 텍스처 없음. 우리 영상은 1080p 30fps 핸드폰 영상: panning 동선, 자동 노출/AF, 모션 블러, 반복 패턴 가능성. G4Splat 의 MASt3R SfM 이 후자의 데이터 분포에 약함 → 잘못된 매칭 → 점 폭증 → OOM. 너 잘못 아님 — 가우시안 분야 공통의 데이터-품질 의존성 문제.

🛠다음 시도 시 적용할 fix (우선순위 순)
